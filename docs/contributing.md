# Contributing / Build process

To build Pathfinder from source: install the PHP dependencies, then compile the front-end assets (JS / SCSS / images) with the Gulp toolchain.

> This page covers **building the application**.  
> For running it (podman/docker) see [deployment.md](deployment.md).

## What gets built

Pathfinder ships two kinds of artifacts, built by two independent toolchains:

| Artifact            | Source            | Tool             | Output                  |
|---------------------|-------------------|------------------|-------------------------|
| PHP autoloader/deps | `composer.json`   | Composer         | `vendor/`               |
| JS bundle           | `js/`             | Gulp (RequireJS) | `public/js/<version>/`  |
| CSS                 | `sass/`           | Gulp (SASS)      | `public/css/<version>/` |
| Images              | `img/`            | Gulp (WebP)      | `public/img/<version>/` |

The sources in `js/`, `sass/` and `img/` are **not** served directly ‚Äî only the compiled output under `public/<type>/<version>/` is.

> **Note**: The compiled assets under `public/{js,css,img}/<version>/` are **not** committed to the repository (they are git-ignored).
> The container image rebuilds them during a dedicated `assets` build stage (see `deployment/pathfinder.Dockerfile`), so a change under `js/`, `sass/` or `img/` ships automatically ‚Äî just rebuild the image.
> You only need to run Gulp yourself for **local development** (file watcher) or to serve the app **without** the container.



## PHP dependencies (Composer)

### Prerequisites

- **PHP ‚â• 7.2 (64-bit)** with extensions: `pdo`, `openssl`, `curl`, `json`, `mbstring`, `ctype`, `gd` (and optionally `redis`).
  See `composer.json` for the authoritative list.
- **Composer** (the image pins 2.1.8, any recent 2.x works).

### Commands

```shell
composer install --no-dev --optimize-autoloader --no-interaction   # production
composer install                                                   # development (includes dev deps)
```

This produces `vendor/` (git-ignored).  
The autoloader namespace `Exodus4D\Pathfinder\` maps to `app/` (PSR-4).
The ESI client library (`Exodus4D\ESI\`) is vendored in-tree at `app/Lib/Esi/`.



## Building frontend assets (Gulp)

### Prerequisites

- **Node.js 24.x + npm** (`package.json` ‚Üí `engines`). The container build uses `node:24`.
  CSS is compiled with `sass` (Dart Sass, pure JS) ‚Äî no native binding, no Node-version lock.
- **GraphicsMagick or ImageMagick** ‚Äî required by the image tasks (`gulp-image-resize`), and must be on `PATH`.
  Only needed if you run the `images` task.

### Commands

Install the Node toolchain (dev-dependencies only - the app has no runtime npm deps):
```shell
npm install
```

Gulp is invoked through the npm script wrapper. Pass Gulp options after a `--` separator:
```shell
npm run gulp <task> -- [--options]
npm run gulp help # list tasks & options
```

### Tasks

| Task         | Purpose                                                                                                                  |
|--------------|--------------------------------------------------------------------------------------------------------------------------|
| `default`    | **Development** - Copies raw JS, builds CSS with sourcemaps, then **watches** for changes. No uglify/compression.        |
| `production` | **Production** - Concatenates + uglifies JS, builds CSS, generates sourcemaps and pre-compressed `.gz` / `.br` variants. |
| `images`     | Convert/crop/resize `img/` sources to WebP + JPG.                                                                        |
| `help`       | Print task & option help.                                                                                                |

Typical use:
```shell
npm run gulp default    # local dev with file watcher
npm run gulp production # one-shot optimized build before committing assets
```

> **Note:** The `.gz` / `.br` compressed variants are git-ignored (`.gitignore`) ‚Äî they're regenerated by the web server / build and not committed.

### The version / build tag

The output folder is the **version tag**, read from `PATHFINDER.VERSION` in `app/pathfinder.ini` (currently `v2.2.4`).
It drives the asset URL (`public/js/<version>/app.js`) and acts as the CSS/JS cache-buster.
Templates reference it via `@PATHFINDER.VERSION`.

By default the tag is `PATHFINDER.VERSION`, but you can override per-build with `--tag`:
```shell
npm run gulp production -- --tag="v2.3.0"
```

- When releasing a change to front-end code, **bump `VERSION`** in `app/pathfinder.ini`.
- The image's `assets` stage then compiles into the new `public/<type>/<version>/` folder on the next build ‚Äî no asset files to commit.
- For a non-container deployment, run a `production` build yourself so `public/` is refreshed.

### Per-build option overrides

Each `default` / `production` preset can be overridden with flags. The table shows each
flag's default value per preset:

| Preset       | `--jsUglify` | `--jsSourcemaps` | `--cssSourcemaps` | `--jsGzip` | `--cssGzip` | `--jsBrotli` | `--cssBrotli` | `--imgActive` |
|--------------|:------------:|:----------------:|:-----------------:|:----------:|:-----------:|:------------:|:-------------:|:-------------:|
| `default`    |    false     |      false       |       false       |   false    |    false    |    false     |     false     |     true      |
| `production` |     true     |       true       |       true        |    true    |    true     |     true     |     true      |     true      |

Other option: `--debug`. See `npm run gulp help` for the full list.



## Linting

### PHP linting

No dedicated PHP linter (phpcs / php-cs-fixer) is configured. Use PHP's built-in syntax checker:
```shell
php -l app/path/to/File.php                                              # single file
find app/ -name "*.php" -exec php -l {} \; | grep -v "No syntax errors"  # all files
```
... or in a container:
```shell
podman run --rm -v "$(pwd):/app:ro" php:8.5-fpm-alpine php -l /app/app/path/to/File.php
podman run --rm -v "$(pwd):/app:ro" php:8.5-fpm-alpine sh -c 'find /app/app -name "*.php" -exec php -l {} \; | grep -v "No syntax errors"'
```

### JS linting

The `default` / watch flow runs ESLint automatically. To run it standalone:
```shell
npx eslint .              # lint all JS sources (except js/lib)
npx eslint js/app/foo.js  # lint a single file
```
... or use a container:
```shell
podman run --rm -v "$(pwd):/app" -w /app node:24-bullseye-slim npx eslint js/app/
```
